1. Introduction

Powered Shopping is a domain-specific AI chatbot application for e-commerce product discovery and cart management. It enables users to search products, get recommendations, add/remove items from the cart, and simulate checkout using voice or typed natural language commands. The project is built as a full-stack web application with a React/Vite frontend and an Express backend.

The key goal is to make shopping actions feel conversational and fast for demos, while still staying reliable. Therefore, the system includes multiple fallbacks (backup product catalog, deterministic intent parser, optional cloud cart persistence) so the app continues to function even in restricted environments.

2. Objectives of the Assessment

This assessment aims to evaluate the ability to solve a real-world problem using Generative AI. In this project we:

  • Identified a meaningful problem: shopping navigation and cart tasks are repetitive and non-conversational.
  • Applied Generative AI to a domain: e-commerce assistant behavior with safe, action-oriented responses.
  • Demonstrated API-based AI integration: model calls are performed via backend APIs with tool-calling.
  • Communicated design choices: architecture, parameters, fallbacks, and deployment are documented.

3. Overview of the Project

The project follows a client-server structure. The client provides authentication, UI views (shop/cart/checkout), and voice/manual command input. The server provides catalog search/filtering, recommendations, cart operations, and AI endpoints for assistant chat and speech transcription.

Frontend responsibilities

  • Firebase Authentication (email/password + Google sign-in)
  • Voice commands (Web Speech API) + typed command fallback
  • Shopping dashboard UI (Shop, Cart, Checkout, Assistant, Settings)

Backend responsibilities

  • Product catalog API with timeout + local JSON fallback
  • Trending/similar recommendation logic
  • Cart persistence (Firestore when configured, otherwise in-memory)
  • AI assistant tool-calling and transcription endpoints

3.1 Domain-Specific AI Chatbot (E-commerce)

A domain-specific chatbot focuses on a constrained domain and solves a narrow set of tasks reliably. Powered Shopping is constrained to shopping intents such as search/filter, recommendations, cart updates, and checkout. The assistant is instructed to use tools for actions and avoid inventing product data or IDs.

6. Data Collection and Domain Knowledge Preparation

The assistant’s behavior is grounded using a structured catalog/cart state and consistent prompts. The project uses a reliable data pipeline: remote product catalog when available, with local fallback to ensure demo stability.

6.1 Sources of Data Collection

  • Product catalog from FakeStore API (with local mock catalog fallback for reliability).
  • Domain-specific intent examples (search, add/remove cart, trending/similar recommendations, checkout).
  • UI flows and constraints documented for consistent assistant actions.

6.2 Purpose of Data Collection

  • Provide structured product/cart state to ground assistant responses.
  • Enable consistent prompt instructions and reduce hallucinations.
  • Ensure demo success even if external services are slow/unavailable.

6.3 Integration of Domain Knowledge

  • System prompt enforces “shopping assistant” behavior and prohibits inventing product IDs/prices.
  • Tool-calling maps language to safe actions: search, recommend, add/remove cart, checkout.
  • Fallback intent parser covers common commands when AI is offline.

7. Model Configuration and Parameter Tuning

The assistant is configured for reliable actions and minimal randomness to avoid unsafe shopping decisions. Parameter choices are aligned to the e-commerce domain (precision over creativity).

7.1 Temperature Parameter

Temperature controls randomness. We use a low value (0.2) for stable, deterministic behavior suitable for shopping operations and tool-calling.

7.2 Top-p (Nucleus Sampling)

Top-p controls probability mass sampling. It is not explicitly set in code (API default applies). For a shopping assistant, leaving top-p at default is acceptable when temperature is low and responses are grounded by tool results.
